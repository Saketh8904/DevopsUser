# Load Testing Report

## 1. Objective

The objective of this load test was to evaluate the performance, reliability, and scalability of the deployed Node.js application hosted on an AWS EC2 Ubuntu instance. The test measured application behavior under increasing concurrent user load and identified potential bottlenecks.

---

## 2. Tooling & Method

- **Tool:** k6
- **Application:** Node.js + Express
- **Web Server:** Nginx Reverse Proxy
- **Platform:** AWS EC2 (Ubuntu 26.04 LTS, t2.micro)
- **Region:** ap-south-1
- **Target Endpoint:** http://13.206.69.42/
- **Load Profile:**
  - 0 → 10 Virtual Users (30 seconds)
  - 10 → 50 Virtual Users (1 minute)
  - 50 → 100 Virtual Users (1 minute)
  - 100 → 0 Virtual Users (30 seconds)
- **Monitoring Tools:**
  - Amazon CloudWatch
  - CloudWatch Dashboard
  - CloudWatch Agent
  - Linux commands (`top`, `free -m`, `vmstat`)

## 7. Observations

- The application successfully handled concurrent requests throughout the test.
- CPU utilization increased proportionally with the number of virtual users.
- Memory utilization remained stable because the Node.js application consumed relatively low memory.
- Network traffic increased steadily during higher loads.
- Response time increased as concurrent users increased.
- No EC2 status check failures were observed.
- CloudWatch metrics matched the load generated by k6.
- The application remained available throughout the test.

---

## 8. Performance Bottleneck Analysis

### Primary Bottleneck

The primary bottleneck observed was CPU utilization on the AWS Free Tier t2.micro instance.

Because the instance has limited vCPU resources and burstable CPU credits, response times increased when CPU utilization became high.

### Secondary Factors

- Memory usage remained within acceptable limits.
- Network throughput was not saturated.
- Disk utilization remained low.
- No storage bottlenecks were observed.

---

## 9. Recommendations

1. Upgrade from **t2.micro** to **t3.small** or **t3.medium** for production workloads.

2. Configure an **Application Load Balancer (ALB)** with an **Auto Scaling Group**.

3. Enable **CloudFront** to cache static assets.

4. Optimize Nginx configuration.

5. Enable HTTP Keep-Alive.

6. Add Redis or another caching layer for frequently accessed data.

7. Configure additional CloudWatch alarms for:
   - CPU Utilization
   - Memory Utilization
   - High Response Time
   - HTTP 5xx Errors

8. Use multiple EC2 instances behind a Load Balancer for high availability.

---

## 10. Conclusion

The deployed Node.js application successfully handled increasing traffic on an AWS Free Tier EC2 instance.

CloudWatch monitoring confirmed stable memory usage and expected increases in CPU utilization during peak load. Performance remained acceptable under moderate traffic; however, sustained high concurrency is limited by the compute capacity of a single t2.micro instance.

For production deployments, implementing horizontal scaling, load balancing, and caching would significantly improve performance, reliability, and fault tolerance.
